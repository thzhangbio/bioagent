# 医学编辑 AI 员工 — 产品蓝图

> 版本：v0.1
> 日期：2026-04-01
> 状态：规划阶段

---

## 一、总纲（三条原则）

1. **把简单交给用户，把复杂留给自己**
2. **用户可以不专业，我们一定要专业**
3. **我们的产品是一个可以完成该岗位 80% 任务的类真人员工**

---

## 二、战略定位

### 不做什么
- 不做"泛内容生成器"
- 不做填表单、选模板、点按钮的工具型产品
- 不做需要独立安装和学习的 App

### 做什么
- 做一个**医学营销内容转化引擎**，伪装成一个飞书上的同事
- 用户加一个飞书好友 = 雇了一个医学新媒体编辑
- 用户怎么跟真人员工交代任务，就怎么跟它交代任务

### 核心差异化
- **医学专业兜底**：所有营销话术在专业上立得住脚，不踩雷。这是普通营销 AI 和医学外行无法提供的核心价值
- **转化导向**：不只生成"好内容"，而是生成"能带来客户的内容"。从"小编的替代者"变成"营销效果的责任人"
- **类真人体验**：有记忆、能成长、会主动、能改稿，不是一次性工具

### 长期路线（四阶段）

| 阶段 | 聚类 | 定位 | 说明 |
|------|------|------|------|
| 当前 | D-新媒体/内容运营 | 现金牛 | AI 可替代性最高，见效最快，错误容忍度高 |
| 同步启动 | B-学术论文/SCI编辑 | 技术壁垒 | D 类赚现金流，B 类构建长期护城河 |
| 后续 | C-医学推广/会务支持 | 扩展市场 | 物料初稿可 AI 化，客户交互保留给人 |
| 灯塔 | A-临床医学写作 | 终极认证 | 专业壁垒最高，需要足够信任积累后再触碰 |

## 三、产品形态

### 载体：飞书
- 产品以飞书机器人（Bot）形态存在
- 用户使用方式：添加飞书好友 → 对话
- 不需要任何独立 UI、不需要安装、不需要培训

### 为什么是飞书
1. **零学习成本**："加好友→发消息"每个职场人都会，产品形态本身就在教育用户正确的使用方式
2. **完整的能力生态**：飞书文档（写稿）、多维表格（排期/数据）、知识库（记忆）、日历（提醒）、任务（追踪）——覆盖一个白领员工的几乎所有数字化工作场景
3. **轻量化交付**：不需要客户 IT 部门介入，小型医疗公司（0-100人）最在意这一点

### 交互原则
- 纯对话，自然语言输入
- 结构化卡片输出（不是一大坨文字）
- 不花哨、不撒娇、不人设——它是员工，不是朋友
- 参考产品：ADHD 管理器的飞书交互模式

### 技术基座：飞书 CLI
- 飞书官方开源 CLI 工具，原生支持 Cursor
- 200+ 命令，11 个业务域，19 个 AI Agent Skills，2,500+ API 端点
- 三层架构：Shortcuts（快捷命令）→ API Commands（精确控制）→ Raw API（全量接口）
- CLI 就是 AI 员工的"手"——它能做飞书上人类员工能做的几乎所有事情

## 四、骨架架构

### 用户视角

```
用户的飞书
├── [聊天窗口]  ←→  AI 员工（对话交互）
├── [飞书文档]  ←→  AI 员工（写稿/改稿）
├── [多维表格]  ←→  AI 员工（排期/数据）
├── [知识库]    ←→  AI 员工（记忆/学习）
└── [日历]      ←→  AI 员工（提醒/排程）
```

### 技术视角

```
用户飞书消息
      │
      │ 飞书 CLI (lark-event WebSocket)
      ▼
┌─ 我们的 Agent 后端 ─────────────────┐
│                                      │
│  [意图理解] → [任务编排] → [工具调用] │
│   (LLM)       (工作流)    (飞书CLI)  │
│      ↑                               │
│  [医学知识]    [记忆系统]             │

```

用户看到的只有上半部分——飞书里的那个聊天好友。
所有的"复杂"都在下半部分。

## 五、三个里程碑

### 里程碑 1：能说话

**目标**：用户觉得"它听得懂我说什么"

#### 三层能力

**耳朵 — 接收消息**
- 飞书开放平台自建应用，配置消息接收权限
- `lark-event` WebSocket 监听 `im.message.receive_v1`
- 提取：谁发的、发的什么、在哪个对话里

**大脑 — 理解意图**

AI 员工不是通用聊天机器人，它有明确的岗位职责。用户说的话分为五类：

| 意图 | 举例 | AI 行为 |
|------|------|--------|
| 布置任务 | "帮我写一篇关于玻尿酸的小红书" | 进入内容创作工作流 |
| 提供信息 | "我们主要做医美产品" | 存入记忆系统 |
| 修改反馈 | "第二段太生硬了" | 定位上次交付物并修改 |
| 询问建议 | "最近写什么话题好？" | 进入选题策划工作流 |
| 岗位外请求 | "帮我订个机票" | 礼貌拒绝，说明不在职责范围内 |

核心是一个 System Prompt，定义了"岗位说明书"——它是一个医学新媒体编辑，不是万能助手。

**嘴巴 — 回复消息**

不回复一大段文字。用飞书交互卡片（Interactive Card）做结构化回复：
- 选题推荐 → 卡片列表，每个有按钮可选
- 确认类 → 简洁卡片，一键操作
- 所有复杂性隐藏在按钮背后

#### 验证标准

5 句真实的"老板对员工说的话"：
1. "帮我写一篇关于我们新产品的小红书笔记" → 正确识别为内容创作任务
2. "最近写什么话题好" → 正确识别为选题建议需求
3. "我们是做医美的，主打玻尿酸填充" → 正确识别为公司信息，存入记忆
4. "上次那篇改一下，开头太平了" → 正确识别为修改反馈
5. "你觉得这个标题怎么样：'3天见效的抗衰秘密'" → 识别为审核需求，指出"3天见效"可能违反广告法

5 句都过 = 里程碑 1 达标。

### 里程碑 2：能干活

**目标**：用户觉得"它写的东西我基本能直接用"

#### 交付位置设计

| 场景 | 交付位置 | 理由 |
|------|---------|------|
| 简短回复 | 聊天窗口 | 快速、轻量 |
| 内容创作 | 飞书文档 | 可编辑、可批注、可导出 |
| 排期计划/数据 | 飞书多维表格 | 结构化、可协作 |
| 时间节点 | 飞书日历 | 融合现有工作习惯 |

#### 核心工作流（以"写一篇小红书笔记"为例）

```
用户："帮我写一篇关于我们玻尿酸新品的小红书笔记"
                 ↓
      AI 内部处理（用户不可见）：
      1. 调取记忆 → 公司画像、产品卖点、目标人群
      2. 医学检索 → 玻尿酸专业信息，确保站得住脚
      3. 平台适配 → 小红书内容规则、爆款特征
      4. 合规审查 → 广告法合规检查
      5. 转化设计 → 在合适位置埋转化钩子
      6. 生成完整内容
                 ↓
      创建飞书文档，写入内容
                 ↓
      聊天窗口发送卡片：
      [✅ 小红书笔记已完成]
      [📄打开文档] [🔄重写] [💬有想法]
```

#### 改稿：两条路径

**路径 A — 在聊天里说**
- 用户："开头太平了，要有冲击力"
- AI 修改文档中对应段落 → 聊天里发对比卡片

**路径 B — 在文档里批注**
- 用户在飞书文档中划线批注
- AI 读取批注 → 理解意图 → 修改该段落 → 回复批注
- 这是杀手级体验：跟真人员工的协作方式一模一样

#### "我们一定要专业"的三个落地点

**1. 医学兜底**
- 用户说"写产品能祛斑"→ AI 自动降级为"有助于改善色素沉着"
- 主动告知用户修改原因

**2. 合规检查**
- 每篇内容交付前自动检查：绝对化用语、医疗效果承诺、处方药宣传
- 交付卡片标注"已通过合规检查"，有风险项则列出

**3. 转化设计**
- 标题用爆款结构
- 正文自然植入产品信息
- 结尾设计行动引导
- 附带标签、发布时间等运营建议

#### 验证标准

1. 写一篇小红书笔记 → 风格对、医学准、有转化、合规
2. 同话题写公众号长文 → 风格完全不同，不是简单改写
3. 用户在文档批注"改一下" → AI 精确修改，不是重写全文

### 里程碑 3：能成长

**目标**：用户觉得"越用越离不开"

#### 成长的三个维度

**维度一：对公司的了解越来越深**

每次交互自动提取信息：
- 任务交代中的公司/产品信息 → 补充公司画像
- 用户的选题偏好（选了什么、否决了什么）→ 记录偏好
- 改稿模式（总是把什么改成什么）→ 学习调性
- 用户反馈"这篇效果好" → 标记优秀案例

存储方式：
- `lark-base` 多维表格：结构化记录，用户可查看和修改
- `lark-wiki` 知识库：品牌知识和优秀案例沉淀

**维度二：对行业的感知越来越敏锐**

AI 主动推送有价值的信息：
- 发现行业热点 → 发送话题提醒卡片，建议是否跟进
- 频率控制：一周一两次，每次言之有物，不是骚扰

**维度三：对工作节奏的把控越来越稳**

- 每周自动生成内容排期计划（基于历史发布节奏和数据表现）
- 用户审批后，AI 按排期自动执行
- 用户的操作从"布置任务"简化为"审稿→批准"

对应飞书能力：
- `lark-calendar`：排期同步到日历
- `lark-task`：每篇内容作为任务追踪（待写→初稿→待审→已发布）
- `lark-im`：定时发送计划和提醒

#### 成长的终极形态

用了 3 个月后，AI 的输出不再是"通用的好"，而是"对这家公司特别好"。同一个话题给不同公司写出来风格完全不同——因为它记住了每家公司的调性和老板的审美。

**它不是在变聪明，是在变成"你的人"。**

#### 验证标准

1. 记忆验证：用户第 1 天说的信息，第 30 天不用重复，AI 自动使用
2. 主动性验证：AI 主动推送的话题建议，用户觉得"说得有道理"
3. 自运转验证：AI 生成的周计划，用户基本只需微调

## 七、商业模式

- **定价对标**：不对标 AI 工具（几十元/月），对标人力成本（一个医学编辑月薪 8-15K + 五险一金）
- **收费方式**：订阅制（按月）或按 Token 用量计费
- **交付方式**：用户在飞书上添加好友即完成交付，零部署成本
