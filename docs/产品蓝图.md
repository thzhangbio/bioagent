## 〇、当前阶段规划（极简 RVP · 战略共识）

**若砍掉一切功能，只留一个**：用户像交代真人一样扔进去一条**写作要求**，系统产出一份**能直接用的**内容（非泛泛闲聊）。

**往回推的必然结构**：「能用」依赖 **懂平台与调性、懂医学表达、懂可核对的专业事实、懂该客户的偏好与素材** → 对应 **多集合可检索知识库**（平台调性、医学/合规片段、**精选文献事实层**、个人上传；**可选**微信长文风格语料作表达参考）→ **有库、能检索、再生成**，比堆功能优先。分层与集合命名见《知识库分层与文献库规划.md》。

**本阶段刻意收窄的边界（节省带宽）**

| 方向 | 策略 |
|------|------|
| **部署与规模** | **个人/本机能跑通即可**；不追求多人并发、不追求极限负载与高可用架构。 |
| **多租户** | **不作为本阶段工程目标**；代码上可用「集合/来源」区分三类库与个人素材，**物理多租户隔离可后移**。 |
| **合规成本产品化** | 不做完整合规产品（审计台、对外 SLA）；内容侧保留**基础规则与 RAG 医学片段**即可，不把「合规体系」做成里程碑负担。 |

**本阶段必须保留的底线（最低成本实现）**

| 方向 | 策略 |
|------|------|
| **数据安全与隐私（尤其客户资料）** | **不可与「极简」一并扔掉**：密钥与凭证不入库、不入 Git；客户上传与向量落盘路径可控（`.gitignore`、本机目录权限）；**调用大模型/Embedding 时对客户原文的日志与留存最小化**；必要时对静态文件做磁盘级或应用级防护（按实现选型逐步加）。目标是 **少数据、少暴露、少泄露面**，而非一上来做等保。**可勾选验收表**见《实现路线图》**任务 2.0 — 隐私底线 · 验收勾选**（五项，里程碑 2 必过）。 |

**向量与基础设施选型**：优先 **本机可运行、依赖少、易回退**（如嵌入式向量库或单机 Postgres）；后续若商业化，可自本 Git 节点起重构，**不因假想规模提前绑死架构**。

**与第四条核心原则的关系**：长期仍坚持 **用户专属知识库与多租户级隔离** 作为产品形态；**当前里程碑**先完成 **单用户闭环 + 三类库 + 隐私底线**，再演进到规模化。

## 一、总纲（三条原则）

1. **把简单交给用户，把复杂留给自己**
2. **用户可以不专业，我们一定要专业**
3. **我们的产品是一个可以完成该岗位 80% 任务的类真人员工**

### 核心原则（第四条，长期不变）

4. **知识库与 RAG 是产品内核，不是附属模块**
   - **必须实现**：平台调性/运营参考、专业知识参考、工作流增强，都依赖**可检索的知识**；生成路径默认为 **「检索 → 再生成」**，而非纯模型幻觉。
   - **商业壁垒**：壁垒不仅在于模型或飞书集成，更在于 **RAG 算法、分块与元数据策略、以及可持续积累的结构化与非结构化语料**（含我方预置库、**期刊级文献清洗稿**与客户侧语料）；**文献事实与营销语感须分集合治理**，避免混检成「单一真理」。
   - **用户专属知识库（长期形态）**：必须允许客户**上传并构建属于自己的知识库**；规模化后通过 **租户隔离** 保证客户之间上下文互不可见。与「通用写作 AI」的本质差异在于 **可检索的专业与客户上下文**。

## 二、战略定位

### 不做什么
- 不做"泛内容生成器"
- 不做填表单、选模板、点按钮的工具型产品
- 不做需要独立安装和学习的 App

### 做什么
- 做一个**医学营销内容转化引擎**，伪装成一个飞书上的同事
- 用户加一个飞书好友 = 雇了一个医学新媒体编辑
- 用户怎么跟真人员工交代任务，就怎么跟它交代任务

### 核心差异化
- **可检索的专业性（RAG）**：调性库、医学与合规片段、**文献事实库（期刊全文清洗稿等）**、用户上传资料，**可选**叠加 **微信医学长文风格库（仅结构与语气参考，非事实依据）**，经 **Embedding + 向量检索** 进入生成链路；交付物可标注引用来源或依据类型。这是与「只会聊天的通用大模型」的分水岭。
- **医学专业兜底**：在检索与规则双重约束下，营销话术在专业上立得住脚，不踩雷。
- **转化导向**：不只生成"好内容"，而是生成"能带来客户的内容"。从"小编的替代者"变成"营销效果的责任人"
- **类真人体验**：有记忆、能成长、会主动、能改稿，不是一次性工具
- **个人/客户知识库**：上传与偏好经同一 RAG 管道进入检索；长期目标为每租户独立语料与索引，**当前 RVP 以单用户验证可用性为主**。

### 长期路线（四阶段）

| 阶段 | 聚类 | 定位 | 说明 |
|------|------|------|------|
| 当前 | D-新媒体/内容运营 | 现金牛 | AI 可替代性最高，见效最快，错误容忍度高 |
| 同步启动 | B-学术论文/SCI编辑 | 技术壁垒 | D 类赚现金流，B 类构建长期护城河 |
| 后续 | C-医学推广/会务支持 | 扩展市场 | 物料初稿可 AI 化，客户交互保留给人 |
| 灯塔 | A-临床医学写作 | 终极认证 | 专业壁垒最高，需要足够信任积累后再触碰 |

## 三、产品形态

### 载体：飞书
- 产品以飞书机器人（Bot）形态存在
- 用户使用方式：添加飞书好友 → 对话
- 不需要任何独立 UI、不需要安装、不需要培训

### 为什么是飞书
1. **零学习成本**："加好友→发消息"每个职场人都会，产品形态本身就在教育用户正确的使用方式
2. **完整的能力生态**：飞书文档（写稿）、多维表格（排期/数据）、知识库（记忆）、日历（提醒）、任务（追踪）——覆盖一个白领员工的几乎所有数字化工作场景
3. **轻量化交付**：不需要客户 IT 部门介入，小型医疗公司（0-100人）最在意这一点

### 交互原则
- 纯对话，自然语言输入
- 结构化卡片输出（不是一大坨文字）
- 不花哨、不撒娇、不人设——它是员工，不是朋友
- 参考产品：ADHD 管理器的飞书交互模式

### 技术基座
- **飞书**：机器人、文档、多维表格、知识库、日历、任务等通过 **OpenAPI / Node SDK** 集成（开发与自动化以 SDK 为主；CLI 可作本地辅助）。
- **大模型**：Claude 等，承担理解与生成；**不负责**长期记忆与精确事实检索——该职责由 **RAG** 承担。
- **RAG 子系统**：解析与分块 → 向量化 → 向量存储（**当前阶段本机/单部署优先**，统一 `rag-store.json` 多 `collection`）→ 检索与重排序 → 注入 Prompt；用户上传文件经同一管道进入 **`personal`**；**期刊文献**经 **`ingest:literature` 进入 `literature`**；**微信风格语料**进入 **`wechat_style`**（块级区分良医汇/梅斯等子风格，**默认可不参与**对话检索，由产品开关或创作场景显式启用）。

## 四、骨架架构

### 用户视角

```
用户的飞书
├── [聊天窗口]  ←→  AI 员工（对话交互）
├── [飞书文档]  ←→  AI 员工（写稿/改稿）
├── [多维表格]  ←→  AI 员工（排期/数据）
├── [知识库]    ←→  AI 员工（记忆/学习）
└── [日历]      ←→  AI 员工（提醒/排程）
```

### 技术视角

```
用户飞书消息
      │
      │ WebSocket / 长连接（自建应用）
      ▼
┌─ Agent 后端 ─────────────────────────────────────────────┐
│                                                           │
│  [意图理解] → [检索 RAG] → [任务编排] → [工具调用]        │
│   (LLM)   调性/专业/文献事实/用户库/可选微信风格  (工作流)   (飞书 API)    │
│      ↑              ↑                ↑                  │
│  [合规规则]    [向量库·分集合]    [结构化记忆]          │
│                                                           │
└───────────────────────────────────────────────────────────┘
```

用户只看到飞书里的聊天与文档；**检索与知识积累**是专业性与壁垒所在，必须在架构上显式存在，而非隐含在 Prompt 里。

## 五、三个里程碑

### 里程碑 1：能说话

**目标**：用户觉得"它听得懂我说什么"

#### 三层能力

**耳朵 — 接收消息**
- 飞书开放平台自建应用，配置消息接收权限
- `@larksuiteoapi/node-sdk` 长连接（`WSClient`）监听 `im.message.receive_v1`
- 提取：谁发的、发的什么、在哪个对话里

**大脑 — 理解意图**

AI 员工不是通用聊天机器人，它有明确的岗位职责。用户说的话分为五类：

| 意图 | 举例 | AI 行为 |
|------|------|--------|
| 布置任务 | "帮我写一篇关于玻尿酸的小红书" | 进入内容创作工作流 |
| 提供信息 | "我们主要做医美产品" | 存入记忆系统 |
| 修改反馈 | "第二段太生硬了" | 定位上次交付物并修改 |
| 询问建议 | "最近写什么话题好？" | 进入选题策划工作流 |
| 岗位外请求 | "帮我订个机票" | 礼貌拒绝，说明不在职责范围内 |

核心是一个 System Prompt，定义了"岗位说明书"——它是一个医学新媒体编辑，不是万能助手。

**嘴巴 — 回复消息**

不回复一大段文字。用飞书交互卡片（Interactive Card）做结构化回复：
- 选题推荐 → 卡片列表，每个有按钮可选
- 确认类 → 简洁卡片，一键操作
- 所有复杂性隐藏在按钮背后

#### 验证标准

5 句真实的"老板对员工说的话"：
1. "帮我写一篇关于我们新产品的小红书笔记" → 正确识别为内容创作任务
2. "最近写什么话题好" → 正确识别为选题建议需求
3. "我们是做医美的，主打玻尿酸填充" → 正确识别为公司信息，存入记忆
4. "上次那篇改一下，开头太平了" → 正确识别为修改反馈
5. "你觉得这个标题怎么样：'3天见效的抗衰秘密'" → 识别为审核需求，指出"3天见效"可能违反广告法

5 句都过 = 里程碑 1 达标。

### 里程碑 2：能干活 + 有据可依（含 RAG v1）

**目标**：用户觉得"它写的东西我基本能直接用"，且**能感知内容有据可查、有品牌与医学约束**（非空泛生成）。

**与核心原则的关系**：里程碑 2 **必须**交付可用的 **RAG 最小闭环**——至少包含：**多集合可检索**（平台调性、医学/专业、个人上传与偏好；**已落地** **文献事实层 `literature`**；**微信风格层 `wechat_style` 为可选增强**，不阻塞闭环）、**生成与改稿前检索注入**、**我方预置库与文献库可逐步加厚**。**单用户本机可用**即可验收多平台与文档交付；**多租户隔离**不纳入本里程碑必达项。**客户资料须满足「〇、当前阶段规划」中的隐私底线**（密钥、落盘、日志最小化）。合规以 **规则 + RAG 片段** 为主，不单独做「合规产品化」。不再将「第一次可用的知识检索」推迟到仅里程碑 3。

#### 交付位置设计

| 场景 | 交付位置 | 理由 |
|------|---------|------|
| 简短回复 | 聊天窗口 | 快速、轻量 |
| 内容创作 | 飞书文档 | 可编辑、可批注、可导出 |
| 排期计划/数据 | 飞书多维表格 | 结构化、可协作 |
| 时间节点 | 飞书日历 | 融合现有工作习惯 |

#### 核心工作流（以"写一篇小红书笔记"为例）

```
用户："帮我写一篇关于我们玻尿酸新品的小红书笔记"
                 ↓
      AI 内部处理（用户不可见）：
      1. 调取记忆 → 公司画像、产品卖点、目标人群
      2. RAG 检索 → 用户专属库 + 调性/运营库 + 专业/合规片段 + **文献事实（若已灌库）** + **可选微信风格语料（若启用；仅结构/语气，不当作临床定论）**（按查询与集合/元数据过滤）
      3. 平台适配 → 小红书内容规则、爆款特征（可与检索结果融合）
      4. 合规审查 → 规则扫描 + 检索到的合规依据（可引用）
      5. 转化设计 → 在合适位置埋转化钩子
      6. 生成完整内容（须能说明主要依据来自哪些检索片段或规则）
                 ↓
      创建飞书文档，写入内容
                 ↓
      聊天窗口发送卡片：
      [✅ 小红书笔记已完成]
      [📄打开文档] [🔄重写] [💬有想法]
```

**用户上传**：在里程碑 2 内需打通至少一种入口（如私聊上传文件 → 解析分块入库），使「专属知识库」可被检索使用。

#### 改稿：两条路径

**路径 A — 在聊天里说**
- 用户："开头太平了，要有冲击力"
- AI 修改文档中对应段落 → 聊天里发对比卡片

**路径 B — 在文档里批注**
- 用户在飞书文档中划线批注
- AI 读取批注 → 理解意图 → 修改该段落 → 回复批注
- 这是杀手级体验：跟真人员工的协作方式一模一样

#### "我们一定要专业"的三个落地点

**1. 医学兜底**
- 用户说"写产品能祛斑"→ AI 自动降级为"有助于改善色素沉着"
- 主动告知用户修改原因

**2. 合规检查**
- 每篇内容交付前自动检查：绝对化用语、医疗效果承诺、处方药宣传
- 交付卡片标注"已通过合规检查"，有风险项则列出

**3. 转化设计**
- 标题用爆款结构
- 正文自然植入产品信息
- 结尾设计行动引导
- 附带标签、发布时间等运营建议

#### 验证标准

1. 写一篇小红书笔记 → 风格对、医学准、有转化、合规；且 **检索链路可观测**（日志或交付说明中可见使用了知识库）
2. 同话题写公众号长文 → 风格完全不同，不是简单改写
3. 用户在文档批注"改一下" → AI 精确修改，不是重写全文
4. **用户上传资料** → 进入其专属库后，后续生成能**显式利用**其中内容（而非仅当附件存在）

### 里程碑 3：能成长（运营化记忆 + 飞书深度集成）

**目标**：用户觉得"越用越离不开"

**与 RAG 的关系**：里程碑 3 **不重复发明「第一次 RAG」**，而是在里程碑 2 的向量与 **集合模型** 之上，做 **记忆与知识的运营化、飞书原生载体同步、主动触达与排期**；若产品进入多客户阶段，再在此之上强化 **租户隔离与合规运营**。**飞书多维表格 / Wiki** 作为 **结构化协作与内容沉淀界面**，与 **向量知识库** 为 **互补关系**：可同步、可双写，**向量库负责语义检索，Base/Wiki 负责协作与审计**。

#### 成长的三个维度

**维度一：对公司的了解越来越深**

每次交互自动提取信息：
- 任务交代中的公司/产品信息 → 补充公司画像
- 用户的选题偏好（选了什么、否决了什么）→ 记录偏好
- 改稿模式（总是把什么改成什么）→ 学习调性
- 用户反馈"这篇效果好" → 标记优秀案例；**可选将优质段落回灌入向量库（需审核策略）**

存储方式：
- `lark-base` 多维表格：结构化记录，用户可查看和修改
- `lark-wiki` 知识库：品牌与案例沉淀；**可与 RAG 入库管道同步**，形成「人可编辑 + 机器可检索」

**维度二：对行业的感知越来越敏锐**

AI 主动推送有价值的信息：
- 发现行业热点 → 发送话题提醒卡片，建议是否跟进
- 频率控制：一周一两次，每次言之有物，不是骚扰

**维度三：对工作节奏的把控越来越稳**

- 每周自动生成内容排期计划（基于历史发布节奏和数据表现）
- 用户审批后，AI 按排期自动执行
- 用户的操作从"布置任务"简化为"审稿→批准"

对应飞书能力：
- `lark-calendar`：排期同步到日历
- `lark-task`：每篇内容作为任务追踪（待写→初稿→待审→已发布）
- `lark-im`：定时发送计划和提醒

#### 成长的终极形态

用了 3 个月后，AI 的输出不再是"通用的好"，而是"对这家公司特别好"。同一个话题给不同公司写出来风格完全不同——因为它记住了每家公司的调性和老板的审美。

**它不是在变聪明，是在变成"你的人"。**

#### 验证标准

1. 记忆验证：用户第 1 天说的信息，第 30 天不用重复，AI 自动使用
2. 主动性验证：AI 主动推送的话题建议，用户觉得"说得有道理"
3. 自运转验证：AI 生成的周计划，用户基本只需微调

## 七、商业模式
