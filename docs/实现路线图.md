| 维度 | 选择 | 理由 |
|------|------|------|
| 语言 | TypeScript | 类型安全，飞书 CLI 通过 npm 安装，生态一致 |

## 里程碑 1：能说话

### 前置准备

**步骤 1：飞书开放平台配置**
- 登录 https://open.feishu.cn，创建自建应用
- 获取 App ID 和 App Secret
- 开启机器人能力
- 配置权限：im:message（收发消息）、im:chat（群聊管理）
- 配置事件订阅：im.message.receive_v1

**步骤 2：本地环境搭建**
- 安装飞书 CLI：`npm install -g @larksuite/cli`
- 安装 AI Skills：`npx skills add larksuite/cli -y -g`
- 配置认证：`lark-cli config init`（输入 App ID / App Secret）
- 登录：`lark-cli auth login --recommend`
- 验证：`lark-cli doctor`

**步骤 3：项目初始化**
- `pnpm init`
- 安装依赖：typescript、@anthropic-ai/sdk（Claude API）
- 配置 tsconfig.json
- 配置 .env（FEISHU_APP_ID、FEISHU_APP_SECRET、CLAUDE_API_KEY）

### 开发任务

**任务 1.1：消息接收**
- 用 lark-event 的 WebSocket 模式监听飞书消息事件
- 解析消息体：提取发送人、消息内容、对话 ID
- 目标：在终端能看到用户发来的每条消息

**任务 1.2：意图识别**
- 编写 System Prompt（岗位说明书）：定义 AI 的角色、能力范围、回复风格
- 将用户消息 + System Prompt 发给 Claude API
- Claude 返回：意图分类 + 回复内容
- 五类意图：布置任务 / 提供信息 / 修改反馈 / 询问建议 / 岗位外请求

**任务 1.3：消息回复**
- 将 Claude 的回复通过 lark-cli im +messages-send 发回给用户
- 先实现纯文本回复，跑通闭环
- 然后升级为飞书交互卡片（Interactive Card）回复

**任务 1.4：入职对话**
- 首次对话时，AI 主动提问了解公司信息
- 对话式收集：公司名称、产品、目标客户、品牌调性、竞品
- 将收集到的信息存入本地 JSON 文件（里程碑 1 简化方案，里程碑 3 迁移到飞书 Base）

### 验证标准

5 句测试：
1. "帮我写一篇关于我们新产品的小红书笔记" → 识别为内容创作任务
2. "最近写什么话题好" → 识别为选题建议
3. "我们是做医美的，主打玻尿酸填充" → 识别为公司信息，存入记忆
4. "上次那篇改一下，开头太平了" → 识别为修改反馈
5. "'3天见效的抗衰秘密'这个标题行吗" → 识别为审核，指出广告法风险

## 里程碑 2：能干活

### 前置条件
- 里程碑 1 全部完成
- 飞书应用增加权限：docx:document（文档读写）

### 开发任务

**任务 2.1：内容创作工作流**
- 用户说"写一篇XX" → router 分发到 content-create workflow
- Workflow 步骤：
  1. 从记忆系统读取公司画像
  2. 根据目标平台确定内容风格模板
  3. 调用 Claude 生成内容（System Prompt 包含：平台规则 + 医学准确性要求 + 转化设计要求）
  4. 合规检查（compliance.ts 扫描禁用词和违规表述）
  5. 用 lark-cli docs +create 创建飞书文档，写入内容
  6. 用 lark-cli im +messages-send 发送卡片通知（含文档链接）

**任务 2.2：多平台适配**
- 同一话题，不同平台输出完全不同的内容
- 平台风格模板：小红书（短句、emoji、hook标题）/ 公众号（深度、专业、引用文献）/ 知乎（问答体、论据充分）
- 模板作为 prompt 的一部分注入，不需要用户指定细节

**任务 2.3：改稿（聊天路径）**
- 用户在聊天中说"改一下XX" → router 识别为修改意图
- Agent 定位最近一次交付的文档
- 调用 Claude 理解修改意图 + 执行修改
- 用 lark-cli docs 更新文档内容
- 回复对比卡片

**任务 2.4：改稿（文档批注路径）**
- 监听飞书文档评论事件（可能需要 lark-event 或轮询）
- 读取批注内容和批注位置
- 调用 Claude 理解批注意图
- 修改文档对应段落
- 在批注下回复"已修改"

**任务 2.5：合规检查模块**
- 内置合规规则库：
  - 广告法禁用绝对化用语清单
  - 医疗广告违规表述清单
  - 化妆品宣称禁用词清单
- 每篇内容交付前自动扫描
- 发现问题：自动修正 + 在交付卡片中说明修改理由

### 验证标准

1. "写一篇关于玻尿酸新品的小红书笔记" → 飞书文档交付，内容风格对、医学准、有转化设计、合规
2. "这个话题再写一篇公众号长文" → 风格完全不同，不是改写
3. 在文档中批注"这段改口语化" → AI 精确修改该段落

## 里程碑 3：能成长

### 前置条件
- 里程碑 2 全部完成
- 飞书应用增加权限：bitable:table（多维表格）、wiki:node（知识库）、calendar:event（日历）、task:task（任务）

### 开发任务

**任务 3.1：记忆系统（飞书 Base）**
- 将里程碑 1 的本地 JSON 迁移到飞书多维表格
- 表 1 — 公司画像：公司名、产品、目标客户、品牌调性、竞品
- 表 2 — 交互记录：日期、任务类型、用户反馈、是否采纳、修改模式
- 表 3 — 优秀案例：标题、内容摘要、用户评价、平台
- 每次交互后，Agent 自动提取关键信息写入对应表
- 生成内容前，Agent 自动读取相关记录作为上下文

**任务 3.2：主动建议**
- 定时任务（如每周一早上 9:00）
- Agent 检索行业热点（通过联网搜索或预设信息源）
- 结合公司画像判断相关性
- 如果有值得跟进的话题，主动发送建议卡片
- 频率控制：每周不超过 2 次

**任务 3.3：内容排期**
- 用户说"帮我排一下这个月的计划" → Agent 生成排期方案
- 用 lark-cli base 写入多维表格（日期、选题、平台、状态）
- 用 lark-cli calendar 同步到飞书日历
- 用 lark-cli task 创建对应任务卡片
- 到期自动提醒 + 开始写初稿

**任务 3.4：周报/复盘**
- 每周五自动汇总本周完成的内容
- 输出：完成了几篇、哪些已发布、用户反馈统计
- 建议：下周重点方向
- 以飞书卡片消息发送

### 验证标准

1. 第 1 天说的公司信息，第 30 天 AI 自动使用，不用重复
2. AI 主动发的话题建议，用户觉得有道理
3. AI 生成的周计划，用户基本只需微调
