## 技术选型

| 维度 | 选择 | 理由 |
|------|------|------|
| 语言 | TypeScript | 类型安全，与飞书 SDK、前后端工具链一致 |
| 运行时 | Node.js | 18+ 推荐；飞书 SDK 与异步 IO 适合长连接与批处理 |
| 飞书交互 | `@larksuiteoapi/node-sdk` | 长连接事件、发消息、文档等统一身份；CLI 仅作本地调试辅助 |
| LLM | Claude API（可经中转） | 理解与生成；**事实与调性以 RAG 为准** |
| **向量与 RAG** | **Embedding API + 向量库（如 PGVector / 云向量 / Milvus 等择一）** | **多租户语义检索、用户上传语料、预置库运营**；具体实现在里程碑 2 锁定 |
| 包管理 | pnpm | 快速、节省磁盘 |

## 里程碑 1：能说话

### 前置准备

**步骤 1：飞书开放平台配置**
- 登录 https://open.feishu.cn，创建自建应用
- 获取 App ID 和 App Secret
- 开启机器人能力
- 配置权限：im:message（收发消息）、im:chat（群聊管理）
- 配置事件订阅：im.message.receive_v1

**步骤 2：本地环境搭建**
- 安装飞书 CLI：`npm install -g @larksuite/cli`
- 安装 AI Skills：`npx skills add larksuite/cli -y -g`
- 配置认证：`lark-cli config init`（输入 App ID / App Secret）
- 登录：`lark-cli auth login --recommend`
- 验证：`lark-cli doctor`

**步骤 3：项目初始化**
- `pnpm init`
- 安装依赖：typescript、`@larksuiteoapi/node-sdk`、`@anthropic-ai/sdk` 等
- 配置 tsconfig.json
- 配置 .env（`FEISHU_APP_ID`、`FEISHU_APP_SECRET`、`ANTHROPIC_API_KEY` 等，见 `.env.example`）

### 开发任务

**任务 1.1：消息接收**
- 用 `@larksuiteoapi/node-sdk` 的 `WSClient` 长连接监听 `im.message.receive_v1`
- 解析消息体：提取发送人、消息内容、对话 ID
- 目标：在终端能看到用户发来的每条消息（与当前仓库 `src/lark/events.ts` 一致）

**任务 1.2：意图识别**
- 编写 System Prompt（岗位说明书）：定义 AI 的角色、能力范围、回复风格
- 将用户消息 + System Prompt 发给 Claude API
- Claude 返回：意图分类 + 回复内容
- 五类意图：布置任务 / 提供信息 / 修改反馈 / 询问建议 / 岗位外请求

**任务 1.3：消息回复**
- 将回复通过飞书 API（`im.v1.message` create/reply）发回用户
- 先实现纯文本回复，跑通闭环
- 然后升级为飞书交互卡片（Interactive Card）回复

**任务 1.4：入职对话**
- 首次对话时，AI 主动提问了解公司信息
- 对话式收集：公司名称、产品、目标客户、品牌调性、竞品
- 将收集到的信息存入本地 JSON 文件（里程碑 1 简化方案，里程碑 3 迁移到飞书 Base）

### 验证标准

5 句测试：
1. "帮我写一篇关于我们新产品的小红书笔记" → 识别为内容创作任务
2. "最近写什么话题好" → 识别为选题建议
3. "我们是做医美的，主打玻尿酸填充" → 识别为公司信息，存入记忆
4. "上次那篇改一下，开头太平了" → 识别为修改反馈
5. "'3天见效的抗衰秘密'这个标题行吗" → 识别为审核，指出广告法风险

## 里程碑 2：能干活 + 有据可依（RAG v1）

> **原则对齐（产品蓝图）**：本里程碑必须交付 **RAG 最小闭环** + **用户专属可上传知识库**；与「只会聊天的通用模型」拉开差距。飞书文档交付、多平台、合规、改稿与 RAG **同一阶段完成**，不将「首次可用检索」推迟到仅里程碑 3。

### 前置条件
- 里程碑 1 全部完成
- 飞书应用增加权限：**docx:document**（云文档创建/编辑）；后续按需增加 **drive**（若从云空间拉文件）
- 可选：卡片相关 **cardkit** 权限（交付通知用交互卡片）

### 开发任务

**任务 2.0：RAG 基础设施（必须先于或与 2.1 并行落地）**
- **向量库与多租户**：按 `tenant / user_open_id` 隔离；元数据包含 `source`（user_upload | tone_ops | medical_compliance 等）。
- **入库管道**：文件解析（PDF/文本）→ 分块 → Embedding → 写入；**至少一种用户入口**（如私聊上传附件触发异步入库）。
- **检索接口**：`retrieve(query, { collections, topK })` → 拼接进 content-create / 改稿 Prompt；可记录检索日志便于审计与调优。
- **预置库**：调性/运营参考、专业与合规片段（可先小后大，但必须可检索）。

**任务 2.1：内容创作工作流**
- 用户说"写一篇XX" → router 分发到 `content-create` workflow
- Workflow 步骤：
  1. 从记忆系统读取公司画像
  2. **RAG 检索**（用户库 + 预置库，按平台与主题构造 query）
  3. 根据目标平台确定内容风格模板，与检索结果一并注入
  4. 调用 Claude 生成（须约束：优先依据检索片段，避免无引用编造）
  5. 合规检查（`compliance.ts` + 可选合规类检索片段）
  6. 使用 **飞书 Node SDK** 创建云文档并写入正文
  7. 发送通知（文本链或交互卡片，含文档链接；可注明「已参考知识库」类摘要）

**任务 2.2：多平台适配**
- 同一话题，不同平台输出完全不同的内容
- 平台风格模板：小红书 / 公众号 / 知乎；**检索 query 随平台变化**（不同库权重可配置）
- 模板作为 prompt 的一部分注入

**任务 2.3：改稿（聊天路径）**
- 用户在聊天中说"改一下XX" → router 识别为修改意图
- Agent 定位最近一次交付的文档；**改稿前对修改意图 + 段落主题做 RAG 检索**（用户禁忌词、术语表等）
- 调用 Claude 修改；使用 SDK 更新云文档
- 回复对比说明或卡片

**任务 2.4：改稿（文档批注路径）**
- 订阅/监听文档评论相关能力（以开放平台当前事件为准）
- 读取批注内容与位置 → **可选检索** → 修改对应段落 → 回复批注

**任务 2.5：合规检查模块**
- 内置规则库（广告法绝对化、医疗违规表述、化妆品宣称等）
- 每篇交付前扫描；可与 **合规类向量片段** 联合输出「依据」
- 自动修正时说明理由

### 验证标准

1. "写一篇关于玻尿酸新品的小红书笔记" → 飞书文档交付；风格、医学表述、转化与合规达标；且 **检索链路可观测**（日志或交付说明）
2. "这个话题再写一篇公众号长文" → 风格明显不同，非简单改写
3. 在文档中批注"这段改口语化" → 精确改段落
4. **用户上传资料** → 进入专属库后，后续生成 **能利用** 其中内容（可通过针对性命题验收）

## 里程碑 3：能成长

### 前置条件
- 里程碑 2 全部完成（含 **RAG v1 与用户专属库**）
- 飞书应用增加权限：bitable:table（多维表格）、wiki:node（知识库）、calendar:event（日历）、task:task（任务）

### 开发任务

**任务 3.1：记忆与协作载体（飞书 Base / Wiki）+ 与 RAG 协同**
- 将里程碑 1 起的本地结构化记忆 **迁移或同步** 至飞书多维表格（公司画像、交互记录、优秀案例等表结构同原规划）
- **Wiki / 云文档**：作为品牌与案例的**人工可编辑沉淀**；**可选同步管道**：Wiki/指定文件夹变更 → 触发 **重新分块入库**，保持向量库与飞书内容一致（策略：全量/增量由实现阶段定）
- 生成前：除 **向量检索** 外，**按需读取 Base 行级字段**（如强约束字段），与 RAG 结果合并为上下文
- **不替代** 里程碑 2 已交付的向量库；本任务侧重 **运营、协作与审计**，与 RAG **互补**

**任务 3.2：主动建议**
- 定时任务（如每周一早上 9:00）
- Agent 检索行业热点（通过联网搜索或预设信息源）
- 结合公司画像判断相关性
- 如果有值得跟进的话题，主动发送建议卡片
- 频率控制：每周不超过 2 次

**任务 3.3：内容排期**
- 用户说"帮我排一下这个月的计划" → Agent 生成排期方案
- 用 **OpenAPI/SDK** 写入多维表格（日期、选题、平台、状态）
- 同步飞书日历与任务（同上）
- 到期自动提醒 + 开始写初稿（可结合 RAG 选题建议）

**任务 3.4：周报/复盘**
- 每周五自动汇总本周完成的内容
- 输出：完成了几篇、哪些已发布、用户反馈统计
- 建议：下周重点方向
- 以飞书卡片消息发送

### 验证标准

1. 第 1 天说的公司信息，第 30 天 AI 自动使用，不用重复（含 Base 与/或向量侧记忆）
2. AI 主动发的话题建议，用户觉得有道理（可结合 RAG 与热点源）
3. AI 生成的周计划，用户基本只需微调

## 实施顺序总结

```
第 1 步：飞书开放平台创建应用 + 本地环境搭建
第 2 步：项目初始化（TypeScript + 依赖）
第 3 步：任务 1.1 — 能收到消息
第 4 步：任务 1.2 — 能理解意图
第 5 步：任务 1.3 — 能回复消息（先文本，再卡片）
第 6 步：任务 1.4 — 入职对话 + 基础记忆
  → 里程碑 1 验证 ✓

第 7 步：任务 2.0 — RAG 基础设施（向量库、入库、检索、多租户、至少一种上传入口）
第 8 步：任务 2.5 — 合规检查模块（可与 2.0 并行）
第 9 步：任务 2.1 — 内容创作工作流（生成前必须 retrieve）
第 10 步：任务 2.2 — 多平台适配
第 11 步：任务 2.3 — 聊天改稿
第 12 步：任务 2.4 — 文档批注改稿
  → 里程碑 2 验证 ✓

第 13 步：任务 3.1 — Base/Wiki 与 RAG 协同、记忆运营化
第 14 步：任务 3.2 — 主动建议
第 15 步：任务 3.3 — 内容排期
第 16 步：任务 3.4 — 周报/复盘
  → 里程碑 3 验证 ✓
```

**说明**：里程碑 2 中 **2.0 与 2.1 强耦合**——验收「能干活」时须同时验收「有据可依」；顺序上可先搭好最小向量链路与空库检索，再接入真实用户上传与预置库加厚。
