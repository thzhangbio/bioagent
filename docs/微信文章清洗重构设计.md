# 微信文章清洗重构设计

> 日期：2026-04-06
> 状态：进行中
> 目标：将 `side-tools/wechat-article-cleanup/` 重构为与 `pdf-text-cleanup/` 同风格的金字塔式目录，显式支持公众号来源、文章类别、结构块处理三层规则

---

## 1. 设计目标

当前 `wechat-article-cleanup` 的问题不在于“不能清洗”，而在于：

- 主线逻辑集中在 `pipeline.ts` 与 `clean-article.ts`
- 不同公众号的处理策略未成为显式阶段
- 同一公众号下不同文章类别的策略未成为显式阶段
- 文章内部结构块（图注、导流、文献、署名等）虽已有规则，但尚未形成可无限细分的阶段目录

本次重构目标：

1. 采用与 `pdf-text-cleanup` 一致的金字塔目录
2. 让“公众号来源”“文章类别”“结构块处理”成为显式阶段
3. 让旧入口命令继续可用，但内部统一转调到 `pyramid/`
4. 让输出成品带上更丰富的可治理元数据

---

## 2. 三层治理模型

### 2.1 公众号来源层

回答“这是谁的文章”。

示例：

- `medsci`
- `liangyi_hui`
- `generic_wechat`

这一层负责：

- 公众号线路识别
- 默认子风格写入
- 公众号特有尾部、署名、导流、参考文献习惯的规则入口

### 2.2 文章类别层

回答“这篇文章属于哪类内容”。

示例：

- `literature_digest`
- `clinical_news`
- `conference_news`
- `expert_viewpoint`
- `activity_promo`
- `recruitment_or_course`
- `roundup`
- `generic_article`

这一层负责：

- 同一公众号下不同类型文章的差异化处理
- 标题与正文的启发式分类
- 为后续不同类别保留不同清洗方案入口

### 2.3 结构块层

回答“文章内部有哪些块，这些块怎么处理”。

示例：

- `title`
- `lead`
- `body`
- `caption`
- `diversion`
- `references`
- `byline`
- `footer`

这一层负责：

- 正文与非正文区分
- 图注、导流、参考资料、署名、运营尾注等块级规范化
- 后续可针对某类块继续细分规则

## 3. 目标目录

```text
side-tools/wechat-article-cleanup/
├── README.md
├── pipeline.ts                     # 兼容壳 → pyramid 总控
├── cli.ts                          # 兼容壳 → 单文件清洗入口
├── archive-inbox.ts                # 兼容壳
├── archive-out.ts                  # 兼容壳
└── pyramid/
    ├── README.md
    ├── wechat-article-cleanup.ts
    ├── stage-shared.ts
    ├── segment-links-to-inbox/
    ├── segment-inbox-to-out/
    ├── segment-out-to-knowledge/
    └── segment-out-to-archive/
```

主线固定为：

- `links -> inbox`
- `inbox -> out`
- `out -> knowledge`
- `out -> archive`

---

## 4. inbox -> out 细分

真正最关键的是 `segment-inbox-to-out/`，建议先拆成：

```text
segment-inbox-to-out/
├── 00-entry-routing/
├── 01-source-profile/
├── 02-article-category/
├── 03-structure-blocks/
├── 04-markdown-render/
└── 05-write-final/
```

### 00-entry-routing

- 读取 raw HTML
- 提取 source URL、fetchedAt、原始元数据
- 统一生成本篇清洗上下文

### 01-source-profile

- 根据 `mp_name` / HTML 特征识别公众号来源
- 输出 `wechat_source_profile`
- 同时给出 `wechat_style_variant`

### 02-article-category

- 根据标题、meta、正文片段做启发式分类
- 输出 `wechat_article_category`

### 03-structure-blocks

- 从 `#js_content` 拆出结构块
- 标记图注、小标题、导流、文献、署名、尾注等
- 为后续细化规则提供块级语义

### 04-markdown-render

- 将结构块转成最终 Markdown
- 保留块级语义产物

### 05-write-final

- 生成 YAML
- 写入 `out/*.md`
- 重命名 `inbox/*.raw.html` 与 `out` 对齐

---

## 5. 输出元数据升级

建议在 `out/*.md` 的 YAML 中新增：

- `wechat_source_profile`
- `wechat_article_category`
- `wechat_style_variant`
- `kb_wechat_id`
- `title`
- `mp_name`
- `published_at`
- `published_at_cn`
- `url`

这样后续无论是入库、检索还是审计，都能按来源线、类别线、块线做治理。

---

## 6. 重构策略

这次不做“推倒重写”，而做“搬主线 + 提显式阶段”：

1. 先建 `pyramid/` 与四段主线
2. 先把旧逻辑迁进对应阶段
3. 再把 `source profile` / `article category` / `structure blocks` 显式化
4. 最后把旧入口改为兼容壳

---

## 7. 完成判据

重构完成后，应满足：

1. `pnpm run wechat-article-pipeline` 仍可用
2. `pnpm run wechat-article-clean -- <raw.html>` 仍可用
3. 公众号来源识别成为显式阶段
4. 文章类别识别成为显式阶段
5. 结构块处理成为显式阶段
6. 目录可继续无限细分，而根总控不变
