# 知识库分层与文献库规划

> 状态：**分层策略、风险原则与扩展方向**见下文；「文献库」与「**微信风格库 `wechat_style`**」已含 **`pnpm run ingest:wechat`** 与槽位检索。**侧车清洗产出**见 §6.2.1；**槽位窄片段第二条管道**仍可后续加；**句子级引用、页码/模块追溯**等见「远期规划」，实现暂缓。  
> 本文整合团队讨论：明显/不明显问题、真理型与工作型、L1–L4 与正交维度、调用策略与治理。

---

## 〇、已暴露问题（产品体验与风险）

### 明显

| # | 现象 | 可能原因 | 优先级 |
|---|------|----------|--------|
| 1 | **标题乱**（不像真实对外标题） | 从用户句抽主题时量词/结构被吃入，或截断规则不当 | 高：观感与检索 |
| 2 | **文档「在哪」、谁可编辑** | 云文档以应用身份创建时常落在应用目录；分享链默认只读，用户感觉「只有医小编能改」 | 高：协作预期（需在回复/文档中说明位置与权限，必要时调整创建身份或文件夹策略） |
| 3 | **文体不像小红书** | 生成提示偏通用科普；小红书运营库未单独加权/模板 | 中：与 RAG 接入节奏相关，暂可接受 |

### 不明显但长期关键：「软文毒害体」

模型**很顺滑地把向量库里的内容（含客户上传、产品话术）写进正文**，短期像「刚好用上某产品」；长期若**所有库同一权重混检**，各类软文、营销片段会被混成「真理」，削弱**医学专业性与权威感**。

**结论**：不是「有一个知识库就够了」，而要**区分知识性质**，并**控制怎么用**；不能默认「全库同时检索」。

## 一、知识性质：真理型 vs 工作型（原则）

| 类型 | 定位 | 谁维护 | 模型应如何使用 |
|------|------|--------|----------------|
| **类真理型 / 权威专业知识** | 接近可采信的医学事实与共识边界 | 我方录入，偏**权威期刊、指南**等（与下文物献库、`medical` 收紧方向一致） | 可作事实与合规表述的**主要依据**（仍须符合临床习惯与法规） |
| **工作型 / 个性化知识** | 服务某公司/某次任务的材料、话术、产品说明 | 我方或客户上传（`personal` 及未来按客户区分的集合） | **可参考、不可盲从**；风格、卖点、禁忌等**不得默认升格为医学真理** |

**互斥与裁决（讨论共识）**：冲突时倾向 **用户最新明确表述 > 权威医学事实 > 工作型材料**（具体实现见后文「防污染」）。

## 二、分层框架 A：产品向「四层」（与仓库集合对应）

从入库与检索策略上，向量库中的集合按**层级语义**划分（检索时可按层过滤、加权或分通道，**禁止无差别全库一锅炖**）。

| 层级 | 含义 | 当前对应集合 | 说明 |
|------|------|----------------|------|
| **事实层（Factual）** | 可核对来源的客观论述 | **`literature`（文献库）** | 精选期刊等经 `pdf-kb-pipeline` 的 `.kb.md` 切块入库；可扩展 DOI/链接。属「真理型」的**主载体之一**。 |
| **规范层（Normative）** | 合规边界、医学表述红线、平台规则 | **`medical`**（及合规向预置） | 预置 `medical-compliance.md` 等；未来可拆「权威指南摘要」与「泛医学表述」子集。 |
| **表达层（Expression）** | 调性、结构、转化套路，**不是医学真理** | **`platform_tone`** | 小红书/公众号等模板与避雷；**仅风格与结构，不升格为疗效事实**。 |
| **表达层（Expression）·扩展** | 真实账号发表的医学长文**语感与结构**（微信生态等），**不是期刊事实** | **`wechat_style`** | **块级** `wechatStyleVariant`：`liangyi_hui`（良医汇肿瘤资讯系）/ `medsci`（梅斯学术系）等；检索时可传 `wechatStyleVariants` **只选一种**或**不传**以混合参考；**不得与 `literature` 混为同一真理来源**。 |
| **情境层（Contextual）** | 客户/用户单次材料 | **`personal`** | 飞书上传等；**工作型**，须与事实层区分权重与 Prompt 角色。 |

**未归入四层但保留**

- **`job_post`**：内部实验/岗位数据，默认**不参与**生产对话 RAG（与既有设计一致），或仅显式开关启用。

分层是**逻辑划分**；物理上仍可在 `data/knowledge/rag-store.json` 用 `collection` 区分，未来可演进为分文件/分索引。

## 三、分层框架 B：技术向 L1–L4（与框架 A 对齐）

按**可信度角色**划分，便于与「禁止全库检索」「Prompt 分角色」对齐：

| 层级 | 含义 | 典型内容 | 模型怎么用 |
|------|------|----------|------------|
| **L1 权威医学 / 事实** | 可采信边界 | 指南摘要、精选文献要点、审慎表述 | 事实与合规底线；**`literature` + 收紧后的 `medical`** |
| **L2 平台与运营** | 调性、结构、转化 | 小红书/公众号模板、标题套路；**可选**微信长文风格语料 | **仅风格与结构**，不当作疗效依据 → **`platform_tone`** + **`wechat_style`**（按场景显式启用；子风格见上表） |
| **L3 工作型 / 客户** | 单客户或单次任务材料 | 产品 briefing、话术、上传笔记 | 可参考、须标不确定性 → **`personal`** |
| **L4 内部实验 / 非生产** | 测试数据 | 岗位库等 → **`job_post`** | 默认不注入生产对话 |

**过渡映射（讨论稿）**

- L1 ← `literature` + 逐步将 **`medical` 收紧为权威入库**；或未来新建 `medical_authoritative` 再迁移。  
- L2 ← `platform_tone` + **`wechat_style`**（微信医学长文语感；**metadata** `wechatStyleVariant` 区分账号/风格线）。  
- L3 ← `personal`（未来可按客户/空间细分）。  
- L4 ← `job_post`，默认关闭对话 RAG。

## 四、「知识库参照」与调用策略（产品原则）

- 提供**可选范围**（勾选/多选）：如「仅权威专业」「仅我的工作库」「权威 + 工作」「只要自己的库、不要专业库」等。  
- **禁止**：在用户未选择时，**始终对所有集合同时、等权检索**。  
- **默认策略**需产品拍板：例如默认仅 L1+L2，L3 仅在上传后开启；或每次会话显式确认「本次参照哪些库」。  
- **实现方向**：`retrieve` 传入 `collections[]` 与可选权重；Prompt 内**分块标注**来源层级（真理型 vs 参考型），工作型材料可单独成块并加说明：「以下为客户上传，非医学事实依据」。

## 五、正交维度（头脑风暴，可与 L1–L4 组合）

在「四层 / L1–L4」之外，可用下列维度做**策略矩阵**，避免无限堆集合名。

### 5.1 证据强度（医学事实内部再切）

| 细类 | 检索/生成提示 |
|------|----------------|
| 指南 / 共识 | 优先；冲突以新版指南为准 |
| 教科书级常识 | 背景解释 |
| 前沿 / 单篇研究 | 仅「可能」「研究显示」，禁止当定论 |
| 器械/药品说明书摘要 | 仅事实核对，不作营销扩写 |

### 5.2 合规风险带（横切各层）

同一句话可能医学上成立、广告法/平台仍不允许：可为命中或生成阶段打 `risk_band`（安全区 / 灰区 / 禁区），走不同模板（科普稿 / 审核稿等）。

### 5.3 时效与版本

Stable（稳定常识）vs Versioned（指南年份）vs Ephemeral（活动文案，**不宜进事实层**）。

### 5.4 模态与粒度

词条 / 段落 / 整篇摘要；纯文本 vs 表格（剂量表）vs FAQ；可先结构化命中再向量段落，减少胡编。

### 5.5 主体与责任

Publisher 官方、KOL/投放历史、UGC——与工作库区分「谁对这段话负责」。

### 5.6 管道角色（替代「单次 retrieve+write」）

讨论中的多段流水线：**FactCheck**（仅 L1+说明书摘要）→ **Tone**（L2）→ **ClientContext**（L3，输出带免责）→ **Compliance**（规则/词表）。生成可先事实草稿再套调性再过合规。

### 5.7 场景卡片（降低用户认知负担）

用户选「场景」而非勾选四个库名，例如：科普小红书·非产品 / 含品牌提及（审后发）/ 学术公众号 / 内部培训。每张卡片预绑定：**启用层、权重、合规等级、是否允许 L3**。

### 5.8 治理与防污染

来源标签、入库置信度、互斥裁决规则、对同一客户话术片段的**冷却与配额**，避免全文「像软文」。

## 六、文献库与微信风格库 — 当前工程落地

### 6.1 文献库 `literature`（事实层 / L1 载体之一）

**目标**：把清洗后的论文正文向量化进 **`literature`**，支撑检索；**不承诺**句级溯源（见下文远期）。

**入库**

- 将终稿 Markdown（推荐 `*.kb.md`）放入 `data/knowledge/literature-inbox/`。  
- 执行：`pnpm run ingest:literature`（**仅替换** `literature` 集合）。  
- MinerU 原始稿 → `side-tools/pdf-text-cleanup`（`pnpm run pdf-kb-pipeline`）产出 `*.kb.md` 后，再拷贝至 inbox 灌库。

**DOI 作为同一篇文献的主键（补充与合并语义）**

- 侧车 **`sourceUrl`** 为 `https://doi.org/10....`（或 **`paperId`** 直接写注册 DOI `10.xxxx/...`）时，入库后每条向量的 **`paperId` 一律为规范化后的注册 DOI**（如 `10.1016/j.ccell.2026.03.004`）。  
- **同一 DOI、多个 `.md` 文件**（例如正文 kb + 附录笔记）视为**同一篇论文的多文件补充**：块 id 按文件名区分，检索与展示时可用 `paperId` 聚合；单次 ingest 会**重灌整个 `literature` 集合**，请保证 inbox 内即「当前要保留的文献全集」。  
- 无 DOI 时回退为 **`paperId`（meta）** 或 **文件名 slug**；正文中**首条** `https://doi.org/10....` 可作为无 meta 时的兜底解析（`src/knowledge/doi.ts`）。  
- 预印本与正式刊若 DOI 不同，视为不同文献，不会合并。

**可选 `同名.meta.json`**

```json
{
  "paperId": "10.1016/j.ccell.2026.03.004",
  "sourceUrl": "https://doi.org/10.1016/j.ccell.2026.03.004",
  "sourceLabel": "Kirschstein et al., Cancer Cell 2026（示例）"
}
```

`paperId` 也可仍用人类可读 slug；只要提供 **`sourceUrl`（DOI）** 即可以 DOI 为准覆盖。

### 6.2 微信风格库 `wechat_style`（表达层 / L2 扩展）

**目标**：收录**已脱敏/已授权**的微信公众号医学长文，用于**句式、标题节奏、篇章结构、运营块形态（导流话术位置与口气）、参考文献区与署名行的版式**；**不作为**可核对医学事实来源（事实仍以 **`literature`** / `medical` 为准）。

**与文献库分工（生成侧）**

- **事实与引用内容**：从 **`literature`**（及用户指定的 DOI/论文）检索与编排。  
- **成稿「像哪条线」**：从 **`wechat_style`** 检索，块级 **`wechatStyleVariant`** 区分账号/线路（如梅斯学术系 `medsci`）。  
- **用户指定导流对象**时：由 **Prompt 约定插入槽位**（例如小结之后、参考文献之前）；该段落的**语气与形式**可参考检索到的 **`> [导流]`** 类块；**导流目标文案**由用户/产品给定，不得从语料库抄用未经授权的第三方品牌话术。

#### 6.2.1 侧车处理产物（文件侧，保证入库原料一致）

**流水线位置**：`side-tools/wechat-article-cleanup/`（详见该目录 **`README.md`**）。

| 环节 | 说明 |
|------|------|
| **入口** | `links.txt` → 抓取或手动的 **`inbox/*.raw.html`**（含 HTTP 注释头中的原文 URL）。 |
| **出口** | **`out/{公众号名}+{文章标题}.md`**，与 inbox raw **同基名**；YAML 含 **`kb_wechat_id`**（稳定主键，类比文献 DOI）、`url`、`mp_name`、`published_at` 等。 |
| **正文结构** | 清洗器将非正文块标为 **`> [属性] …`**，属性包括 **`导流`**、**`文献`**、**`参考资料`**、**`署名·撰文`**、**`署名·编辑`**、**`转载说明`**、**`联系方式`**、**`运营`** 等；**图注**为 **`> 图注：…`**。**默认保留**文末运营与导流段——这是**有意保留的表达层语料**，用于学习「插在何处、长什么样」；**不是**事实层依据。 |
| **不做的** | 不在清洗阶段删除导流/署名；不因「降噪」擅自剥除（除非产品另定合规策略）。互动数据（阅读/点赞等）可选 **`--fetch-stats`**，与正文质量无关。 |

**入库前人工**：校对断行/错别字；确认授权与脱敏；再进入灌库流程。

#### 6.2.2 入库方式（当前约定与槽位扩展规划）

**全文入 `wechat_style`（主线）**

- **输入**：上述 **`out/*.md`**（含 front matter + 全文，含 **`[导流]`** 等与正文）。  
- **切块**：按段落/长度策略切分为多条向量；每条必须带 **`wechatStyleVariant`**（如 **`medsci`** / **`liangyi_hui`**），并建议可追溯 **`sourcePath`**、文首 YAML 中的 **`kb_wechat_id`**（便于去重与审计）。  
- **脚本**：**`pnpm run ingest:wechat`**（`scripts/ingest-wechat.ts`），**仅替换** `wechat_style` 集合，与 `ingest:literature` 分列执行；默认目录为 **`side-tools/wechat-article-cleanup/out/`**（可用 **`WECHAT_STYLE_INBOX`** 覆盖）。  
- **事实边界**：Prompt 与产品说明中明确：**不得**将 `wechat_style` 命中内容当作循证结论；医学论述须对齐 **`literature`** 检索结果。

**槽位语义子库（可选演进，与全文库互补）**

为提升「在固定结构位置仿写导流 / 参考文献列表 / 编辑署名」的检索精度，可从同一批 **`out/*.md`** **再导出**窄片段（例如仅 **`> [导流]`** 块、仅文末 **`> [文献]`** 与参考文献区、仅 **`署名·*`** 行），以 **第二条管道** 入库，二选一工程形态：

1. **仍用集合 `wechat_style`**，块级增加元数据 **`wechatContentSlot`**（示例取值：`body` | `diversion` | `references` | `byline` …），检索时 **`wechatStyleVariants` + `wechatContentSlot` 过滤**；或  
2. **新建独立集合**（如 `wechat_slot_style`），专收槽位句，避免与长文正文向量混检——**以最终实现为准**，原则为 **少而清晰、避免无意义碎片化集合**。

衔接提示：槽位块单独成向量时，可能弱化「与上一段的过渡」；可在切块时 **附带 1～2 句上文** 或依赖 Prompt 要求「承接段末结论再写导流」。

#### 6.2.3 子风格与检索（块级字段）

- **`wechatStyleVariant: "liangyi_hui"`**：良医汇肿瘤资讯等线路。  
- **`wechatStyleVariant: "medsci"`**：梅斯学术等线路。  
- 扩展更多枚举时，保持 **单集合 + 元数据**，避免碎片化多个集合名。

**检索**

- `retrieve(..., { collections: ["wechat_style"], ... })`：混合命中全部已灌子风格。  
- 追加 `wechatStyleVariants: ["liangyi_hui"]` 或 `["medsci"]`：**仅**命中对应子风格（未标注 `wechatStyleVariant` 的块不应入库，**灌库时强制打标**）。  
- 检索侧已支持 **`wechatContentSlots`**（见 `retrieve.ts`）；CLI 探针：`pnpm run search:rag -- "…" --only wechat_style --wechat-slots diversion`。  

**默认产品行为**

- 对话与创作 RAG **默认可不包含** `wechat_style`，避免未配置时噪声；需 **「参考微信成稿风格」** 类场景时**显式**并入 `collections` 或「场景卡片」开关。

**当前代码行为**：对话与创作 RAG **已包含 `literature`**；**`wechat_style` 已存在于类型与 `retrieve`，默认未并入** `CHAT_RAG_COLLECTIONS` / `CONTENT_RAG_COLLECTIONS`；**`ingest:wechat` + 块级 `wechatContentSlot` + `retrieve(..., { wechatContentSlots })`** 已落地；**尚未**实现「用户勾选库」——后续需按第四节收紧默认集合并暴露配置。

## 八、工程节奏建议（讨论稿，非排期承诺）

1. **短期**：标题与文档权限的产品说明；对话侧 **默认检索集合可配置**，避免无意识全库。  
2. **中期**：`medical` 收紧或拆分权威子集；`personal` 明确为工作库；创作流按勾选集合检索；Prompt **防污染**分块。  
3. **长期**：多租户/按客户向量分区；审计（本次生成使用的集合与 chunk）支撑「软文从哪来」复盘。
