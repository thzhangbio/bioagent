# 知识库导入器设计

> 日期：2026-04-06
> 状态：草图已定，待按本稿逐步实现
> 目标：把“清洗后入库”从各类侧车与散脚本中抽离，形成统一、可维护、可扩展的导入层

---

## 1. 设计结论

知识库导入相关能力应以**仓库内独立子工具**存在，而不是继续分散在：

- `side-tools/pdf-text-cleanup/`
- `side-tools/wechat-article-cleanup/`
- `scripts/ingest-literature.ts`
- `scripts/ingest-wechat.ts`
- 未来更多来源的临时 ingest 脚本

推荐形态：

```text
bioagent/
├── tools/
│   └── knowledge-importer/
│       ├── README.md
│       ├── cli/
│       └── pyramid/
```

定位：

- `side-tools/*` 负责把原始脏数据清洗为标准中间稿
- `tools/knowledge-importer` 负责把标准中间稿送入知识库
- `src/knowledge/*` 保留运行时检索、向量存储、类型定义等主项目通用能力

一句话边界：

**清洗器解决“原始内容怎么变干净”，导入器解决“干净内容怎么稳定进库”。**

---

## 2. 为什么单独抽工具

当前入库逻辑已经开始分散：

- 文献：`pdf-text-cleanup` 产出 `*.kb.md`，再走 `ingest:literature`
- 微信：`wechat-article-cleanup` 产出 `*.md`，再走 `ingest:wechat`
- 个人资料：运行时附件直接入 `personal`
- 后续高概率还会出现：指南、说明书、FAQ、手工整理笔记、知识卡片

继续按“每种来源一个 ingest 脚本”发展，会出现三个问题：

1. 入库规则无法集中优化
2. 元数据标准无法统一
3. 验证、归档、去重、upsert 行为会越来越不一致

单独抽导入器后，后续优化会收敛到一个地方：

- 文献按章节切块
- 不同来源的 metadata 标准化
- replace / append / upsert 统一治理
- 入库后抽样验证
- 归档与 manifest 记录

---

## 3. 目标与非目标

### 3.1 目标

- 统一不同知识来源的入库入口
- 提供统一的中间数据结构
- 提供统一的 chunk、embedding、store-write、verify 流程
- 支持来源级适配器，而不是来源级散脚本
- 允许后续把 `literature` 从“整集合 replace”演进到“按 paperId upsert”
- 允许后续做“章节感知切块”“分层降权”“结构化字段过滤”

### 3.2 非目标

- 不是现在就重写运行时 RAG 检索层
- 不是现在就替换 `src/knowledge/*` 全部实现
- 不是现在就做多租户隔离
- 不是现在就做复杂任务编排系统

第一阶段只做：**统一入库骨架**。

---

## 4. 架构原则：采用金字塔式目录

导入器在架构上应**参考** `side-tools/pdf-text-cleanup/pyramid/`，而不是做成普通的“按技术层平铺”的工具目录。

原因：

1. 每个大阶段都能独立扩展出很多细分能力
2. 每个阶段都能在自己的根目录总控文件里随时收束
3. 后续当某一步变复杂时，不需要推翻整体结构，只需在该段下继续加 `00`、`01`、`02` 子目录
4. 便于用 README 把每段职责固定下来，降低未来漂移

### 4.1 结构规则

- 每个功能目录的根代码文件，与该目录同名，是该目录的总控
- 有子目录时，根代码文件负责调度子目录
- 无子目录时，根代码文件直接实现本层功能
- 旧入口脚本只做兼容转调，不承载主实现

### 4.2 推荐目录

```text
tools/knowledge-importer/
├── README.md
├── cli/
│   └── knowledge-import.ts
└── pyramid/
    ├── README.md
    ├── knowledge-importer.ts
    ├── stage-shared.ts
    ├── segment-load-to-normalized/
    │   ├── README.md
    │   ├── segment-load-to-normalized.ts
    │   ├── stage-shared.ts
    │   ├── 00-source-routing/
    │   ├── 01-read-input/
    │   ├── 02-parse-frontmatter/
    │   ├── 03-build-import-documents/
    │   └── 04-normalize-metadata/
    ├── segment-normalized-to-chunks/
    │   ├── README.md
    │   ├── segment-normalized-to-chunks.ts
    │   ├── stage-shared.ts
    │   ├── 00-select-chunker/
    │   ├── 01-section-split/
    │   ├── 02-window-chunk/
    │   ├── 03-build-chunk-records/
    │   └── 04-chunk-quality-check/
    ├── segment-chunks-to-store/
    │   ├── README.md
    │   ├── segment-chunks-to-store.ts
    │   ├── stage-shared.ts
    │   ├── 00-pre-write-check/
    │   ├── 01-embed-batches/
    │   ├── 02-write-store/
    │   ├── 03-verify-write/

    └── tools/
        ├── README.md
        ├── import-literature-kb/
        └── import-wechat-style/
```

### 4.3 主线语义

主线建议固定为：

- `load -> normalized`
- `normalized -> chunks`
- `chunks -> store`

这比“按来源分一堆散脚本”更适合统一管理，也比“按纯技术层扁平拆文件”更利于后续无限延展。

### 4.4 各段职责

- `segment-load-to-normalized/`
  - 把不同输入来源统一变成 `ImportDocument`
- `segment-normalized-to-chunks/`
  - 按来源/内容类型选择 chunk 策略，产出标准化 chunk records
- `segment-chunks-to-store/`
  - embedding、写库、验证、写 manifest

### 4.5 为什么不优先用平铺 `src/adapters + src/chunkers + src/store`

平铺结构短期简洁，但中期容易失控：

- 当某个 adapter 变复杂时，细节会不断堆进一个文件
- 当文献切块需要十几步预处理时，没有天然的“分段总控”
- 当写库前后增加校验、状态记录、归档耦合时，平铺目录会迅速变乱

因此导入器应优先采用：

**总控目录 + 分段目录 + 分段下可继续细分的金字塔式结构。**

## 5. 统一数据模型

### 5.1 ImportSource

```ts
export type ImportSource =
  | "literature_kb"
  | "wechat_style"
  | "personal_note"
  | "guideline"
  | "manual";
```

### 5.2 ImportDocument

导入器的第一层统一对象，表示“一个可入库文档”。

```ts
export interface ImportDocument {
  source: ImportSource;
  sourcePath: string;
  sourceId: string;
  title: string;
  collection: KnowledgeCollection;
  body: string;
  metadata: Record<string, string | number | boolean | null | undefined>;
}
```

说明：

- `sourceId` 是该来源内的稳定主键
- 对文献可优先使用 DOI
- 对微信可优先使用 `kb_wechat_id`
- 对手工文档可退化为 slug

### 5.3 ImportChunkRecord

切块后的统一对象。

```ts
export interface ImportChunkRecord {
  id: string;
  source: ImportSource;
  sourceId: string;
  collection: KnowledgeCollection;
  sourcePath: string;
  sourceLabel: string;
  chunkIndex: number;
  text: string;
  metadata: Record<string, string | number | boolean | null | undefined>;
}
```

说明：

- 最终会映射到现有 `TextChunk`
- 但先保留 importer 自己的中间结构，避免一开始就绑死主项目运行时类型

## 6. 第一版命令形态

推荐只做一个统一入口：

```bash
pnpm run knowledge-import -- run --source literature_kb
pnpm run knowledge-import -- run --source wechat_style
```

后续可扩展子命令，但第一版不需要太复杂。

### 6.1 建议的 CLI 参数

```bash
knowledge-import run \
  --source literature_kb \
  --input side-tools/pdf-text-cleanup/out \
  --collection literature \
  --mode replace-collection \
  --verify
```

```bash
knowledge-import run \
  --source wechat_style \
  --input side-tools/wechat-article-cleanup/out \
  --collection wechat_style \
  --mode replace-collection \
  --verify
```

### 6.2 建议保留的执行阶段

- `load`
- `normalize`
- `chunk`
- `embed`
- `write`
- `verify`
- `manifest`

CLI 可以支持：

- `--from-stage <stage>`
- `--to-stage <stage>`
- `--skip-verify`
- `--dry-run`

这样后续调试很方便。

## 7. 第一版适配器

### 7.1 literature-kb source routing

输入：

- `side-tools/pdf-text-cleanup/out/*.kb.md`
- 或 `data/knowledge/literature-inbox/*.md`

职责：

- 读取 front matter 中的 `kb_metadata`
- 抽取 `doi/title/journal/published/abstract`
- 生成稳定 `sourceId`
- 把文档拆成可供 section-aware chunker 使用的正文结构

输出：

- `ImportDocument[]`

### 7.2 wechat-style source routing

输入：

- `side-tools/wechat-article-cleanup/out/*.md`

职责：

- 读取 front matter
- 抽取 `kb_wechat_id/title/mp_name/published_at/wechat_style_variant`
- 保留对槽位切块有帮助的原始正文标记

输出：

- `ImportDocument[]`

## 8. 切块策略建议

### 8.1 文献：从固定窗升级为章节感知

当前问题：

- `scripts/ingest-literature.ts` 直接对整篇 Markdown 固定窗口切块
- front matter、封面、作者单位、版权声明、参考文献容易混入高价值检索结果

第一版建议：

- 先按一级标题或明显 section 分段
- 识别 `abstract/introduction/results/discussion/methods/references/declarations`
- 每个 section 再做局部窗口切块
- 给 chunk 附加 `sectionType`

建议的 `sectionType`：

- `abstract`
- `introduction`
- `results`
- `discussion`
- `methods`
- `references`
- `declarations`
- `front_matter`
- `unknown`

建议默认策略：

- `abstract/results/discussion`：主检索高价值
- `introduction/methods`：保留
- `references/declarations/front_matter`：保留但后续可降权，甚至支持不入主库

### 8.2 微信：继续槽位感知

微信成稿现有思路基本是对的，应迁入统一导入器：

- `body`
- `caption`
- `diversion`
- `references`
- `byline`
- `footer`

后续只需要把这套逻辑从“脚本特例”变成“适配器 + chunker”。

## 9. 写库策略建议

导入器需要显式支持三种写库模式：

### 9.1 replace-collection

适合：

- `platform_tone`
- `medical`
- `wechat_style`
- 当前阶段的 `literature`

语义：

- 替换整个 collection

### 9.2 append

适合：

- `personal`
- 日常补充笔记

语义：

- 只追加，不删旧块

### 9.3 upsert-by-source-id

适合：

- 后续成熟后的 `literature`
- 稳定主键明确的集合

语义：

- 同一 `sourceId` 的旧块先删后加
- 不影响同集合其他文档

第一版落地顺序建议：

1. 保持 `replace-collection` 兼容现状
2. 在导入器内预留 `upsert-by-source-id` 接口
3. 文献稳定后再切到按 DOI upsert

## 10. 验证与 manifest

导入器不应只负责“写进去”，还应负责“确认写进去”。

### 10.1 verify

第一版最小验证：

- 检查 `rag-store.json` 中是否存在本批次 `sourcePath`
- 检查本批次 `sourceId` 数量是否匹配
- 输出导入文件数、切块数、写入后总块数

第二版可增加：

- 抽样 query 做最小检索验证
- 检查 sectionType 分布

### 10.2 manifest

每次导入建议生成一份 manifest，例如：

```json
{
  "importedAt": "2026-04-06T10:00:00.000Z",
  "source": "literature_kb",
  "inputDir": "side-tools/pdf-text-cleanup/out",
  "mode": "replace-collection",
  "collection": "literature",
  "files": [
    {
      "sourcePath": "...",
      "sourceId": "10.1016/j.xxx",
      "title": "..."
    }
  ]
}
```

用途：

- 归档依据
- 审计依据
- 排障依据

## 11. 与现有代码的衔接方式

### 11.1 第一阶段：桥接，不硬切

第一阶段不必直接删除现有脚本，而是做桥接：

- `scripts/ingest-literature.ts` 改为调用 importer 的 `literature_kb` adapter
- `scripts/ingest-wechat.ts` 改为调用 importer 的 `wechat_style` adapter
- `pdf-kb-out-to-knowledge` 的 ingest 阶段改为调用统一 CLI

这样可以保持命令兼容，降低迁移风险。

### 11.2 第二阶段：逐步收口

等 importer 稳定后：

- `scripts/ingest-literature.ts` 可只留兼容壳
- `scripts/ingest-wechat.ts` 可只留兼容壳
- 文档统一改成 importer 入口

## 12. 推荐的实施顺序

### Phase 1：搭骨架

- 建 `tools/knowledge-importer/`
- 定 `types.ts`
- 定 `run-import.ts`
- 定 `registry.ts`
- 提供统一 CLI

### Phase 2：迁文献

- 实现 `literature-kb` adapter
- 实现 `literature-section-aware` chunker
- 接入现有 `embedTextChunks`
- 写回现有 `rag-store.json`

### Phase 3：迁微信

- 实现 `wechat-style` adapter
- 把现有 `wechat-segments` 逻辑迁或桥接进 importer

### Phase 4：统一验证与 manifest

- 完成 verify
- 完成 manifest
- 把 `pdf-kb-out-to-knowledge` 与后续归档都接入统一状态文件
