## 三、`src/` — 运行时源码

**作用**：飞书长连接收消息 →（可选：**交互卡片**）→（目标：**RAG 检索**）→ Claude 生成 → 飞书 HTTP 发消息。  
**当前**：**里程碑 1 对话闭环**已具备；**RAG** 含统一 `rag-store.json`、`retrieve`、**用户上传文件 → `personal`**、**文献 → `literature`**；**文本对话**经 `router` 前会 **按问句检索** `platform_tone` / `medical` / **`literature`** / `personal`（**不含 `job_post`**，避免岗位实验干扰；**不含 `wechat_style`** 除非产品显式并入检索列表）；**离线** `ingest:all` 灌预置 + 岗位；**文献**由 `ingest:literature` 单独灌入并**仅替换** `literature` 集合。

### 3.1 `src/index.ts`

1. `dotenv/config` 加载 `.env`。  
2. `validateEnv()`：要求 `ANTHROPIC_API_KEY`、`FEISHU_APP_ID`、`FEISHU_APP_SECRET`。  
3. `acquireFeishuWsLock()`（`ws-lock.ts`），防本机双开长连接。  
4. `LarkEventBridge`：  
   - **`im.message.receive_v1`**：`messageId` 去重 → **若含附件** → **`handleUserFileMessage`**（`file-upload-handler.ts`）→ 文本回复；**否则** → 若 **`FEISHU_CONTENT_CARD_FORM=1`** 且 **`isContentCreateIntent`** → **`sendContentCreateFormCard`**（下发卡片 + 文字说明）→ `appendConversationMessages` + **`setWriteMergeGate(chatKey, "awaiting_supplement")`** → **不**走 `handleUserMessage`；否则 → **`handleUserMessage`**（`router`）→ 常规对话或 **内容创作工作流**；`replyMessage` / `sendMessage`（正文多为 Markdown 字符串，经 `text` 类型发送）。  
   - **`card.action.trigger`**（需开放平台订阅）：`FEISHU_CONTENT_CARD_FORM=1` 时由 **`handleContentCreateCardEvent`** 处理表单提交与异步成稿。  
5. `bridge.start()` 启动 WebSocket。

### 3.8 核心文件工作逻辑（摘要）

| 模块 | 主路径（逻辑顺序） |
|------|---------------------|
| **`index.ts`** | 校验环境 → WS 锁 → 注册消息/卡片回调 → 去重 → 附件走上传入库；文本且 **卡片开关 + 创作意图** → 发卡片并短路；否则 **`handleUserMessage`** → 回复飞书。 |
| **`router.ts`** | 创作意图 → 合并历史（若闸门允许）→ **`runContentCreateWorkflow`**；否则 RAG 包裹用户句 → 通用 Claude 对话；维护 **`write-merge-gate`** 与 **`extractAndSaveInfo`**。 |
| **`content-create.ts`** | `loadMemory` + RAG 块 + 画像 + 简报信号 → **（可跳过）动笔前闸门** → 成稿模型调用 → **`createDocumentWithPlainText`** → 返回含 doc 链接的 Markdown；失败时回退说明或正文预览。 |
| **`content-create-card-flow.ts`** | 首句：**LLM 动态题** → session → **builder 卡片**；提交：**解析 `form_value`** → 合并原文与结构化摘要 → **`skipAlignmentGates: true`** 成稿 → 删 session → 发 Markdown；回调 **仅 toast + 异步**，防 WS 超时。 |
| **`write-merge-gate.ts`** | 显式状态机，避免「追问后的补充句」与无关闲聊错误合并进成稿请求。 |
| **`docx-document.ts`** | 创建文档、根块、分段写入纯文本、构建 Web URL、（可选）协作者授权；含重试等网络容错（见 **`FEISHU_API_MAX_RETRIES`**）。 |

## 六、`side-tools/pdf-text-cleanup/` — MinerU 论文 → KB Markdown（侧车）

**作用**：与主 bot **解耦**的 TypeScript 工具链：将 MinerU 导出的 **`.md` + `.json`** 规范为 **`*.kb.md`**（RAG/编辑友好），供拷贝至 **`data/knowledge/literature-inbox/`** 后执行 **`ingest:literature`**。**不参与** `tsc` 对 `src/` 的默认编译输出，由 **`pnpm run pdf-kb-pipeline`** / **`pnpm run pdf-cleanup`** 经 `tsx` 调用。**归档分两路**（与微信侧车一致）：① 流水线产出 **out** 并确认后 **`pdf-archive-inbox`** → `archive/processed-mineru/<时间戳>/`（腾空 **`inbox/`** 内 MinerU 源）；② **向量化入库后** **`pdf-archive-out`** → `archive/ingested-out/<时间戳>/`（腾空 **`out/*.kb.md`**）。若 **`archive/`** 根目录仍有旧版平铺的 `.md`/`.json`，用 **`pdf-archive-legacy-flat`** 迁入 **`processed-mineru/`**。详见 **`README.md`**。

| 文件 | 功能 |
|------|------|
| `pipeline.ts` | 端到端：读原始 md+json → 应用 KB 清洗 → 写 **`out/*.kb.md`**。 |
| `mineru-kb.ts` | KB 正文规范化（残余公式、结构噪声等）。 |
| `mineru-json-structure.ts` | MinerU JSON 版面/结构辅助。 |
| `cleanup.ts` | 通用清洗步骤组合。 |
| `raw-to-preliminary.ts` | 原始稿 → 中间稿。 |
| `cli.ts` | **`pdf-cleanup`** 入口。 |
| `archive-inbox.ts` | **`pnpm run pdf-archive-inbox`**（归档 **`inbox/`** 内 MinerU 源，腾空 inbox） |
| `archive-out.ts` | **`pnpm run pdf-archive-out`**（归档已入库的 **`out/*.kb.md`**，腾空 out） |
| `archive-legacy-flat.ts` | **`pnpm run pdf-archive-legacy-flat`**（将 **`archive/`** 根遗留 MinerU 源迁入 **`processed-mineru/`**） |
| `inbox/` | 可选：待跑 pipeline 的 MinerU **`.md` / `.json`**（见 `inbox/README.md`） |
| `out/` | 生成的 **`*.kb.md` 终稿**（可提交或再复制到 `literature-inbox`）。 |
| `archive/` | **`processed-mineru/`**、**`ingested-out/`** 时间戳子目录（见 `archive/README.md`）。 |

### `side-tools/wechat-article-cleanup/` — 微信公众号文章清洗

## 十一、核心数据流

**A. 飞书对话 — 纯文本（默认）**

```
用户文本 → WSClient（events.ts）→ index 去重
        → router（创作意图 → content-create；否则 rag-context + Claude + memory）
        → messages 回复 → 飞书
```

**B. 飞书对话 — 交互卡片（`FEISHU_CONTENT_CARD_FORM=1`）**

```
首句创作意图 → sendContentCreateFormCard（LLM 动态题 + session + interactive 卡片）
             → 用户选表单项 + form_submit
             → card.action.trigger → handleContentCreateCardEvent（toast 立即返回）
             → 异步 runContentCreateWorkflow（skipAlignmentGates）→ sendMessage 成稿结果
```

**C. 用户文件**

```
用户文件 → events 解析 attachment → file-upload-handler（下载 → 抽取文本 → personal 入库）→ 文本回复说明
```

**D. RAG 离线实验（不经过飞书）**

```
rag-presets/*.md → ingest:presets → 合并入 data/knowledge/rag-store.json（platform_tone / medical）
research/…/01-57-*.md → ingest:jobs → 合并入同一 rag-store.json（job_post）
data/knowledge/literature-inbox/*.kb.md → ingest:literature → 仅替换 rag-store.json 内 literature 集合
side-tools 产出 *.kb.md →（人工拷贝）→ literature-inbox → ingest:literature
用户查询 → search:rag / search:jobs → retrieve() → 终端展示命中块
```

**E. 里程碑进展（与路线图对应）**

- **`content-create` 工作流**：检索 →（可选）闸门 → 生成 → **docx** → 链接；**卡片路径**跳过闸门直接成稿。  
- **待扩展**：独立合规扫描 `compliance.ts`、成稿后基于 docx 的修订流、多平台模板权重等（见 `实现路线图.md`）。
